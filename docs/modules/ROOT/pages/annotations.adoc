= Quarkus Zanzibar Permissions

Zanzibar authorization integrates with Quarkus' standard `@PermissionsAllowed` mechanism. Endpoints declare the required
relation using the permission name, and a `ZanzibarPermission` implementation provides the object type and id.

== Usage

[source,java]
----
@PermissionsAllowed(value = "viewer", permission = TenantPermission.class)
public Tenant getTenant(String tenantId) { ... }

public final class TenantPermission extends ZanzibarPermission {
    private final String tenantId;

    public TenantPermission(String name, String tenantId) {
        super(name);
        this.tenantId = tenantId;
    }

    @Override
    public String getObjectType() { return "tenant"; }

    @Override
    public String getObjectId() { return tenantId; }
}
----

== User Type

Some FGA backends require a user type. You can override it in the permission:

[source,java]
----
@Override
public Optional<String> getUserType() {
    return Optional.of("user");
}
----

You can also configure a default user type globally and per resource class. Resolution order is:
permission override > resource default > global default.

[source,properties]
----
quarkus.zanzibar.default-user-type=global
quarkus.zanzibar.default-user-types."com.acme.MyResource".user-type=customer
----

== Check Context

ZanzibarPermission can supply per-endpoint context and contextual tuples. A CDI
`RelationshipCheckContextSupplier` can also contribute global/request-scoped context.
When both are present, permission context overrides supplier context keys and supplier
contextual tuples are applied before permission tuples.
